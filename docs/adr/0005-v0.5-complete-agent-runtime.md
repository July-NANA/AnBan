# ADR-0005: v0.5 Complete Main Agent, Durable Graph, Interaction and Automation Architecture

- Status: Accepted for v0.5 planning; implementation deferred to delivery Issues
- Date: 2026-07-18
- Scope: v0.5 Complete Agent & Graph CLI

## Context

v0.1 proved one fixed, real, durable CLI execution closure. v0.5 must complete the CLI Agent
Runtime without replacing that path or creating provider-, Skill-, integration-, or
acceptance-specific shortcuts.

## Decision

### Main Agent and capability sufficiency

The Main Agent is the highest-priority decision subject. It understands the goal, obtains the
available inventory, evaluates sufficiency, chooses the lowest-complexity reliable and governed
path, executes, observes, evaluates completion, and replans only when needed.

Capability Sufficiency Evaluation covers model direct-answer ability, structured Capabilities,
ready Skills, `process.execute`, MCP Tools, Memory, and sub-agents. Its strategy vocabulary is:

```text
direct_answer
use_capability
activate_skill
use_process
acquire_skill
delegate
clarify
fail
```

A missing Skill is not a gap when bounded terminal operations can complete the task. Skill
acquisition is appropriate only when existing paths require substantial temporary code, a complex
domain workflow, high-risk improvisation, or a low-confidence implementation.

### General physical execution

Program, file, Web, Browser, Git, code, package-management, and ClawHub operations continue through
Skill instructions or Main Agent decisions into `process.execute`. Domain-specific core handlers
such as WeatherHandler, CalendarHandler, GitHandler, BrowserHandler, PDFHandler, or a privileged
ClawHub installer service are prohibited.

### Task graphs and immutable revisions

Simple tasks keep the fixed General Agent graph. Complex tasks follow:

```text
Main Agent -> TaskGraphSpec -> validation -> immutable GraphRevision
           -> LangGraph Builder -> ExecutionRun
```

A structural plan change creates a new GraphRevision; it never overwrites history. Context-only
updates may retain the active revision.

### Durable Runtime

Runtime will support durable Checkpoint, waiting, resume, cancel, asynchronous continuation, and
restart recovery. It must not replay a side effect when the outcome is uncertain. It reuses valid
completed results after governed replanning.

### Interaction

Every external input and result enters through InteractionEnvelope. Correlated input resumes an
eligible Run; uncorrelated input creates a Task and Run. User updates, background Process results,
MCP, sub-agents, Webhooks, and schedules cannot bypass the Interaction Gateway.

### Automation

Automation decides when to start, which input to carry, and whether to create or resume a Run:

```text
Webhook / Cron / Interval -> Interaction Gateway -> Task / Run
                          -> Main Agent / Graph Runtime
```

Schedulers never execute Process or business functions directly. The default overlapping-run
policy is `skip` unless a later authorized ADR changes it.

### Acceptance integrity

Production must not branch on a fixed Prompt, Skill ID, provider, scenario, expected output, test
variable, filename, Run ID, or environment. Built-in fake installation, in-memory restart
simulation, fake MCP/sub-agent execution, direct Cron target calls, Webhook-to-Runtime bypasses,
test-only success switches, mock-success, and fallback-success are prohibited.

## Consequences

The six modules remain fixed. This ADR authorizes the v0.5 delivery direction but creates no
production Port, Handler, Adapter, schema, migration, or Tool in this preparation delivery. Each
concrete architecture extension remains subject to its delivery Issue, tests, and acceptance Gate.
