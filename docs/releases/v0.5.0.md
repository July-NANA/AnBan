# Anban v0.5.0 Release Candidate Notes

Status: release candidate prepared by D34. No `v0.5.0` tag or GitHub Release is created until P2
Gate #71 and Final Gate #73 pass on the exact pushed `anban` SHA and the user-authorized release
closure completes.

## Product closure

v0.5 completes the CLI Agent Runtime inside the original six modules. The Main Agent evaluates
real inventory and sufficiency, uses bounded completion/replanning, and can run multiple Skills and
Capabilities or acquire an ordinary Skill only when existing paths are genuinely insufficient.
Simple work retains the fixed Agent loop; complex work produces a validated `TaskGraphSpec`, an
immutable `GraphRevision`, and topology-independent dynamic LangGraph execution.

PostgreSQL now persists Task/Session Context, graph revisions, structured Node outputs,
Checkpoints, Interaction inbox deliveries, resume correlations, parent/child Runs, schedule
definitions, and schedule occurrences in addition to the v0.1 execution facts. Background Process
work, mid-run updates, MCP, sub-agents, authenticated Webhooks, and Cron/Interval Automation all
enter through ordinary Composition Roots and reconstruct after process or Application restart.

## Migrations and persistence

The linear Alembic chain ends at `0012_schedule_occurrences`. Development and test profiles must
both be at head. Migration acceptance verifies the authoritative table set, relationships,
append-only graph history, Checkpoint correlations, inbox deduplication, Context constraints,
delegation ownership, and unique Schedule occurrence claims inside rollback-safe transactions.

Restart safety depends on PostgreSQL facts and bounded Workspace worker state, never a mock queue or
fallback result. Valid completed graph occurrences are reused only when definition and transitive
inputs remain equivalent. Capability-backed invalidation that would repeat a side effect is
rejected. Schedule lease recovery reuses the same occurrence and Interaction identities; terminal
inbox delivery reconstructs the existing Run.

## Audit and Trace

Audit remains an allowlisted projection of the same ordered Event stream used by Trace. v0.5 adds
the `agent.*`, `context.*`, `graph.*`, `checkpoint.*`, `interaction.*`, `subagent.*`, `webhook.*`,
and `schedule.*` families. Integration-specific facts include `subagent.child_created`,
`webhook.authenticated`, and `schedule.occurrence_dispatched`. Safe metadata contains logical
identities, hashes, counts, decisions, status, and replay flags—not prompts, raw correlations,
Secrets, provider responses, process arguments/output, or physical paths.

## Side-effect and retry policy

There is no fallback success. Runtime never automatically repeats a returned or ambiguous
Capability side effect. Terminal persistence confirms commit state and compensates only confirmed
uncommitted facts. Inbox duplicates reconstruct terminal Runs; changed semantics conflict.
Background recovery consumes the original worker result. Schedule overlap defaults to `skip`,
missed work is `skip` or `catch_up_once`, leases and attempts are bounded, and persistence/Audit
ambiguity stays retry-pending rather than becoming success.

## Known limitations

v0.5 remains CLI-first. It has no React/React Flow product, full Policy Engine, approval workflow,
strong sandbox, program allowlist, network isolation, RBAC, Risk Graph, visual Replay/Fork, SaaS,
or multi-user collaboration. `process.execute` uses the launching OS user's authority. Temporary
interactive chat remains process-local; durable asynchronous work uses Checkpoints and Interaction
correlations instead.
