## Delivered outcomes

v0.1 proved the `interaction`, `core`, `runtime`, `model`, `capability`, and `persistence`
boundaries through one real CLI execution closure. It delivered a fixed General Agent, a real
OpenAI-compatible Model Port, uniform `skill.activate`, general `process.execute`, PostgreSQL,
Task/ExecutionRun/NodeRun, CapabilityInvocation, Artifact, Event, Audit, Trace, CLI execution and
inspection, restart-safe queries, explicit fail-closed behavior, and no Core bypass.

## What worked

- The six-module boundary and Port/Adapter direction kept providers, Skills, storage, and CLI out
  of Core.
- A bounded Agent loop made model turns, Capability calls, repetition, time, and failure explicit.
- Real E2E Gates caught integration and contract problems that deterministic tests could not.
- Audit and Trace derive from authoritative Events, and uncertain side effects are not blindly
  replayed.
- Small Issues, focused commits, Phase Gates, and exact-SHA CI kept acceptance evidence traceable.
- CLI-first delivery proved execution and restart inspection before introducing a visual product.

## Limitations and debt carried to v0.5

- The graph is a fixed single-node General Agent graph; there is no complete Main Agent planner or
  Capability Sufficiency Evaluation.
- The Skill catalog is fixed at Application startup; there is no runtime refresh and continuation.
- There is no TaskGraphSpec, immutable GraphRevision, dynamic graph construction, Checkpoint,
  waiting, resume, cancellation, or restart recovery for non-terminal Runs.
- Interaction is not yet a durable bidirectional gateway for mid-run updates and asynchronous
  results.
- MCP, sub-agents, Automation, Webhook, Cron, Interval, and long-term Memory are absent.
- `process.execute` remains primarily synchronous.

Known non-blocking acceptance variance: A later real-model Gate A execution returned
`model_rejected` while persistence and Trace remained complete. This is not treated as a v0.1
product regression and does not block v0.5 planning. Issue #30 separately corrected an Acceptance
Contract Mismatch without changing Runtime, Persistence, Capability, or Observability behavior.

## Decisions carried forward

- Do not rewrite the verified v0.1 closure; extend it within the same six modules.
- Continue using `process.execute` as the general physical execution channel for program, file,
  network, Browser, Git, package, and code operations.
- A missing matching Skill is not itself a capability gap.
- No Skill, Provider, source, MCP server, Automation trigger, or acceptance scenario receives a
  core bypass.
- Missing real conditions fail explicitly; mock-success and fallback-success remain forbidden.
