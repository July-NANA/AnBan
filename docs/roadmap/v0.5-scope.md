# v0.5 Complete Agent & Graph CLI Scope

## Position and priority

v0.5 is a CLI-first, functionally complete Agent Runtime centered on a complete Main Agent and
durable graph execution. Main Agent contracts and capability sufficiency precede Task Graph,
Interaction, MCP, sub-agent, and Automation work.

## Epics and phases

Functional ownership uses five Epics: Main Agent Core; Task Graph & Durable Runtime; Interaction
Gateway; Capability & Automation Runtime; and Persistence, Audit & Release.

Development order uses three phases:

1. P1 Complete Main Agent
2. P2 Task Graph & Durable Execution
3. P3 Interaction, Automation & Release Closure

Epic owns capability scope; Phase owns real implementation order.

## Included

- Main Agent state, decision, completion, alternative-path, and bounded-replanning behavior.
- Capability inventory, description, sufficiency evaluation, multi-Capability/multi-Skill use,
  runtime Skill refresh, acquisition, and continuation.
- Task and Session context with bounded durable compression.
- TaskGraphSpec, immutable GraphRevision, dynamic LangGraph construction, branch, loop, parallel,
  join, subgraph, and simple/complex routing.
- Background Process execution, Checkpoint, waiting, resume, cancel, restart recovery, mid-run user
  updates, and valid-result reuse.
- InteractionEnvelope routing, durable inbox, deduplication, Human Input, and asynchronous result
  correlation.
- Real MCP discovery/invocation, sub-agent lifecycle, authenticated Webhooks, Cron, Interval, and
  reliable Automation dispatch.
- PostgreSQL facts, migrations, Audit, Trace, CLI inspection, documentation, and release closure.

## Main Agent decision order

The Agent evaluates direct answer, existing structured Capability, ready Skill, `process.execute`,
MCP, Memory, and sub-agent options before acquiring a Skill. It selects from `direct_answer`,
`use_capability`, `activate_skill`, `use_process`, `acquire_skill`, `delegate`, `clarify`, and
`fail`.

Skill acquisition is used only when existing paths would require substantial temporary code,
complex domain workflow, high-risk improvisation, or a low-confidence result. Bounded file,
network, Browser, Git, package, and code work continues through `process.execute` when sufficient.

## Bounded fixed-loop execution

The simple-task path keeps the fixed `START -> general_agent -> END` LangGraph. Before execution,
the ordinary Composition Root records one inventory-backed sufficiency assessment. Every real
Capability result then returns to the same bounded reasoning loop as an ordered observation, so an
explainable failed or unavailable path can lead to a distinct Capability or Skill without creating
a second scheduler or Run.

An identical observed call is never invoked again. Runtime returns a truthful replay-prevented
control observation, preserves the prior side-effect fact, and leaves the model only the remaining
bounded alternatives. Calls with no structured observation fail explicitly. Audit exposes
`agent.sufficiency_assessed` and `agent.observed` with safe decision, order, retry, and side-effect
metadata; it never stores raw prompts or Capability output. Structural replanning and correlated
mid-run user updates remain on the later durable graph and Interaction delivery path.

Assistant final text is a proposal until a separate closed completion assessment reconciles the
original goal, transcript, real Tool Results, ready paths, and remaining budget. A complete
assessment supplies the authoritative final text. An incomplete assessment must select one exact
ready alternative, request clarification, or fail explicitly. Production enforces the next
strategy/target and never treats Skill activation, Memory retention, a successful intermediate,
or narrated intent as original-goal completion. Replans have their own configurable budget
(default 3, hard maximum 8) in addition to model-turn, Capability-call, repeated-call, and total
time limits.

Audit records `agent.completion_assessed`, `agent.replan_decided`,
`agent.clarification_requested`, and `agent.failure_selected` with bounded decisions, counts, and
hashes only. The structured Model request/response and every later Invocation remain in the same
Run Event stream and reconstruct through ordinary PostgreSQL queries after restart.

The production Skill catalog is live within the same Application. Inventory inspection and
activation re-scan package and Workspace roots through the uniform parser, so added, changed, and
removed Skills are observed without replacing the Registry. Successful catalog-backed activation
emits `skill.catalog_refreshed` with only counts and a deterministic digest before
`skill.activated`. Instructions for multiple Skills accumulate only as paired native Tool Results
and fail closed at the shared context bound.

When the assessment proves acquisition is warranted, Audit records
`agent.skill_acquisition_requested` using only bounded reason flags. The Agent must use ordinary
Skills and Capabilities to search and install, activate a Skill that was not ready in the initial
inventory, and continue the original Task in the same Run. A sufficient selected existing path
must be attempted first, so acquisition cannot replace a reasonable Process or other ready path.

## Durable bounded context

The ordinary production Registry includes `memory.context` for Task and Session context. Task
identity is derived from the current Run; Session identity is created and supplied by Runtime, not
accepted from model arguments. Entries retain source classification, conflict/supersession state,
optional expiry, and bounded sensitivity. Secret-classified and configured protected values are
rejected before persistence.

Compression stores a bounded summary plus ordered coverage of original Entry IDs in one
PostgreSQL transaction. Covered raw entries become superseded but are never deleted. Invalid
coverage, limits, or persistence failure leaves authoritative facts unchanged, so restart recovery
does not depend on process memory. The new Audit Events are `context.read`, `context.recorded`,
`context.compressed`, `context.expired`, and `context.operation_failed`; their metadata contains
only allowlisted counts, identities, operation, and hashes. Runtime anti-replay applies to Memory
exactly as to other Capabilities, so a returned side effect is never repeated after an ambiguous
terminal persistence response.

## Graph, interaction, automation, and evidence

Simple work retains the fixed graph. Complex work produces validated TaskGraphSpec and immutable
GraphRevision facts. Structural replanning creates a revision; context-only updates do not.
Checkpoint and recovery are PostgreSQL-backed and cannot rely on process memory.

D23 makes completed graph results occurrence-specific across replanning. Runtime derives a
conservative fixed point from unchanged node definitions, incoming edges, and transitive input
ancestry. Valid NodeRuns replay from durable output; invalidated pure NodeRuns execute as new
occurrences; Capability-backed results are never automatically repeated. Rejected replacement
graphs do not enter the revision chain, while their safe failure fact remains durable and the
waiting Run remains recoverable.

The first background execution increment reuses `process.execute`: `background: true` returns a
non-terminal acceptance only after the real child process is spawned, emits monotonic progress,
and correlates the eventual terminal result and Artifacts to the same system-owned Invocation.
The ordinary Agent can wait for that terminal result in the same execution or use the same durable
Checkpoint path to return waiting state, resume, or cancel. Checkpoint and transition Events survive
fresh database-only queries, and release of the live execution occurs only after the transition
commits. Full service-process exit recovery rebuilds the original Invocation and consumes the
durable worker result without replay. No result is converted to success.

All external input and asynchronous results pass through Interaction. The shared envelope already
distinguishes user messages, supplemental input, asynchronous Capability/MCP/sub-agent results,
Webhook events, and schedule occurrences. Its route distinguishes new work from a request to
resume an eligible Run. Bounded external resume and deduplication keys are distinct from every
system-owned identity; external normalization rejects identity injection, expired/conflicting
keys fail closed, and safe Event projection hashes key values. D22 implements lookup only for
waiting-continuation supplemental input, persists bounded Task Context, and applies governed
context-only or structural replanning before recovery. D26 adds the general PostgreSQL inbox,
hashed protocol deduplication, post-receipt expiry, restart-safe stale-claim recovery, and durable
rejection. D31 adds real HMAC-authenticated HTTP Webhooks for new work and contextual resume through
that same inbox. Authentication failures remain outside persistence because the sender is
untrusted; authenticated semantic failures are durable. D33 claims durable occurrences and submits
them through that same Interaction gateway; schedulers do not execute business work.
Every release scenario requires database, Audit/Trace, Artifact, restart, real-integration, reverse
case, and anti-hardcoding evidence appropriate to its path.

D32 persists immutable Cron and Interval definitions with strict five-field POSIX validation,
IANA timezone/DST calculation, a bounded initial next occurrence, and safe fresh-process CLI
inspection. Definition creation does not create a Run, inbox delivery, or side effect. D33 adds
the real worker, occurrence claims, Interaction dispatch, idempotency, overlap/concurrency, replay,
bounded leases, and missed-run policy through PostgreSQL and the ordinary gateway.

D30 registers `agent.delegate` in the same Registry and starts the same Runtime as an independent
child Task/Run. PostgreSQL records the parent Run/Invocation relationship and bounded depth before
the parent receives non-terminal acceptance. The child retains its own Model, Capability,
Artifact, Audit, and Trace facts. A `SUBAGENT_RESULT` Interaction after Application restart only
signals readiness; the Handler reconstructs the terminal child aggregate and the parent consumes
that authoritative result without replay. Child failure and cancellation remain explicit, and an
active child is cancelled on service shutdown rather than assigned to an imaginary worker.

D25 turns the shared envelope contract into the single routing gateway before the later inbox.
Route meaning is authoritative before input kind: uncorrelated supported user messages create new
work regardless of logical Adapter source, and a supported resume request must resolve an eligible
external correlation. Every created Run persists a safe `interaction.routed` fact. D26 adds
deduplicated durable delivery, D27-D28 add governed updates and async-result readiness, and D31
admits authenticated Webhook events without changing the production execution path. Only the
trusted Schedule Adapter can provide complete occurrence attestations; unattested external
schedule input is durably rejected rather than falling through to Task creation.

## Non-goals and v1.0 boundary

v0.5 excludes React, React Flow, the full Policy Engine, strong container sandboxing, enterprise
approval, RBAC, Risk Graph, visual Replay/Fork, SaaS, and multi-user collaboration. Those governed
visual and enterprise surfaces remain v1.0 work.

## Definition of done

All delivery Issues and three Gates are complete; S01-S12 pass through ordinary production
composition with real required integrations; PostgreSQL, Audit, Trace, restart, negative cases, and
semantic variants prove reusable architecture; exact pushed SHA passes deterministic CI; and no
secret, bypass, mock-success, or fallback-success exists.
