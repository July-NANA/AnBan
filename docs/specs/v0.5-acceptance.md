# v0.5 Release-Blocking Acceptance Specification

## Common evidence contract

Every scenario uses dynamic task data and semantic variants through the ordinary production
Composition Root. Evidence includes Task/Run/Node/Invocation database facts, ordered Audit/Trace,
Artifacts where relevant, explicit failures, and real restart or integration behavior when
declared. Fake, Stub, and Mock may support unit fault injection but never final acceptance.

Every external scenario input uses the same `InteractionEnvelope` vocabulary. Uncorrelated input
requests a new Task; resumable input carries a bounded external resume key, never a system Task,
Run, Session, NodeRun, Invocation, GraphRevision, Checkpoint, Event, or Artifact identity. An
independent deduplication key may identify delivery without changing routing meaning. Malformed,
expired, conflicting, unknown, and ineligible correlation must reject or remain unresolved; none
may silently create or resume work. Audit/Trace projects only logical namespaces and fingerprints.
These contracts alone are S06-S11 foundation, not durable routing or scenario acceptance.

P1 Gate #72 runs `pnpm acceptance:p1`. Its dedicated Main Agent driver generates two ready Skills
and fresh task objects, exercises direct answer, structured Memory, three semantic ready-Skill
variants, three multi-Skill variants, clarification, and three distinct explicit-failure variants
through the ordinary Composition Root, and reconstructs every result with a new query Application.
The existing real Runtime Gate supplies the general Process, Skill acquisition, same-Run
continuation, and fresh post-install execution evidence; its Process case rejects any acquisition
or Skill activation. A Provider rejection, unavailable registry, missing Artifact, or inconsistent
durable fact fails the Gate.

## S01 Existing Skill completes a real task

- Preconditions: a valid ready Skill and its real prerequisites exist.
- Input: three semantic variants of a task the Skill describes.
- Expected path: inventory -> `skill.activate` -> real Capability execution -> completion.
- Forbidden path: source/slug special case or fabricated result.
- Database/Audit/Trace: correlated Run, activation, Invocation, terminal Events, and counts.
- Artifact: real outputs are snapshotted and queryable when the task produces files.
- Restart/integration: query from a new Application; use the real Skill and execution channel.
- Reverse case: invalid Skill or missing prerequisite fails explicitly.
- Anti-hardcoding: random Skill identity/path and changed task object.
- Pass: real result and complete consistent evidence.

## S02 No matching Skill; process.execute is sufficient

- Preconditions: no matching ready Skill; bounded installed programs can complete the task.
- Input: varied file, network, Git, package, or code tasks.
- Expected path: sufficiency evaluation -> `use_process` without Skill search.
- Forbidden path: acquire a Skill merely because no matching slug exists.
- Database/Audit/Trace: Process Invocation and terminal Run counts agree.
- Artifact: declared real outputs persist with logical URIs.
- Restart/integration: new Application queries evidence; real Process is required.
- Reverse case: an unsafe or insufficient Process approach is not improvised.
- Anti-hardcoding: randomized filenames, content, cwd, and phrasing.
- Pass: bounded task completes with no acquisition Event.

## S03 Acquire a Skill and continue the original task

- Preconditions: model, existing Capabilities, ready Skills, Process, MCP, Memory, and delegation
  are demonstrably insufficient; ordinary ClawHub access is available.
- Input: semantic variants requiring a compatible external Skill.
- Expected path: search -> inspect -> install -> catalog refresh -> activate -> resume original Run.
- Forbidden path: fake install, privileged source branch, or replacement Task.
- Database/Audit/Trace: acquisition, `skill.catalog_refreshed`, activation, and continued execution
  remain correlated to the original Task without storing physical paths or instructions in Audit.
- Artifact: installation evidence and task outputs are real, bounded, and separated.
- Restart/integration: new Application discovers installed `SKILL.md`; real registry operation.
- Reverse case: when Process suffices, no search occurs.
- Anti-hardcoding: randomized compatible Skill and task objects.
- Pass: original goal completes after genuine acquisition.

## S04 One task uses multiple Skills

- Preconditions: at least two independently useful ready Skills.
- Input: one goal requiring both domains.
- Expected path: one Run activates and uses multiple Skills in justified order.
- Forbidden path: merged fake Skill or precomputed output.
- Database/Audit/Trace: distinct activation/Invocation facts share the Run.
- Artifact: each output retains producing Invocation correlation.
- Restart/integration: complete aggregate rebuild; real Skills execute.
- Reverse case: redundant Skill activation is avoided.
- Anti-hardcoding: change Skill identities, ordering opportunity, and task phrasing.
- Pass: combined goal and complete correlations succeed.

## S05 Generate and execute a real TaskGraphSpec

- Preconditions: task complexity exceeds the fixed-loop threshold.
- Input: varied tasks requiring different topologies.
- Expected path: Main Agent -> TaskGraphSpec -> validation -> GraphRevision -> builder -> Run.
- Forbidden path: fixed acceptance template or topology-specific builder.
- Database/Audit/Trace: immutable revision, nodes, edges, execution, and terminal facts.
- Artifact: node outputs correlate to graph nodes and Invocations.
- Restart/integration: persisted graph executes/rebuilds after process exit; real LangGraph path.
- Reverse case: simple tasks retain the fixed graph.
- Anti-hardcoding: random node data and at least three unseen topologies.
- Pass: one builder executes branch/loop/parallel variants correctly.

## S06 Mid-run user update and governed replanning

- Preconditions: an active waiting or background Run.
- Input: supplemental user requirement with correlation ID.
- Expected path: Interaction -> existing Run; context-only update retains revision, structural
  update creates a new immutable revision.
- Forbidden path: mutate prior revision or direct Runtime call.
- Database/Audit/Trace: input, decision, revision choice, and reused/invalidated results are clear.
- Artifact: valid completed results remain queryable.
- Restart/integration: update survives service restart through real Interaction.
- Reverse case: unrelated input creates a new Task.
- Anti-hardcoding: vary update timing, wording, and structural impact.
- Pass: governed plan reflects the update without history loss.

D23 evidence distinguishes concrete completed occurrences: unchanged definition/input ancestry is
reused, invalidated pure work is reexecuted under a new NodeRun, and Capability-backed invalidation
that would replay a side effect is durably rejected. A rejected proposal does not link a revision or
discard the waiting Checkpoint.

D25 gateway evidence varies logical Adapter source and Task object while keeping one route
contract. New user work must create a distinct durable Task/Run and one safe `interaction.routed`
Audit fact. Unsupported later-delivery input kinds and unknown resume correlations must fail before
new work is created; this gateway foundation does not claim D26-D28 inbox or delivery behavior.

## S07 Cancel or correct an active operation

- Preconditions: a cancellable active or waiting operation.
- Input: correlated cancel or correction through Interaction.
- Expected path: signal -> safe cancellation/correction -> durable terminal or revised state.
- Forbidden path: process-memory-only flag or fabricated cancellation.
- Database/Audit/Trace: request, propagation, Process state, and terminal facts agree.
- Artifact: partial/uncommitted outputs are not presented as success.
- Restart/integration: terminal state remains after restart; real child process cleanup.
- Reverse case: late duplicate cancellation is idempotent.
- Anti-hardcoding: vary timing and operation type.
- Pass: no orphan execution or false success.

## S08 Background task, Checkpoint, and restart recovery

- Preconditions: durable background work and PostgreSQL are available.
- Input: task that waits across a complete service-process exit.
- Expected path: checkpoint -> waiting -> restart -> load -> resume -> terminal.
- Forbidden path: in-memory cache or blind replay of uncertain side effects.
- Database/Audit/Trace: checkpoints, waiting transitions, continuation, and counts agree.
- Artifact: pre-restart valid results persist and are not duplicated.
- Restart/integration: full process exit is mandatory; real background Process executes.
- Reverse case: ambiguous side effect stops for explicit recovery.
- Anti-hardcoding: random restart point and task object.
- Pass: one logical Run completes without duplicate side effects.

## S09 Dynamic MCP discovery and real protocol invocation

- Preconditions: a real test MCP server with dynamic Tool descriptors.
- Input: tasks chosen after server startup.
- Expected path: inventory/discovery -> structured invocation -> asynchronous result through
  Interaction -> completion.
- Forbidden path: hardcoded Tool, fake protocol response, or direct core bypass.
- Database/Audit/Trace: server/tool identity, Invocation, correlation, and result are durable.
- Artifact: protocol-produced files use ordinary Artifact persistence.
- Restart/integration: real MCP transport and reconnect behavior are exercised.
- Reverse case: unavailable/malformed server fails explicitly.
- Anti-hardcoding: random Tool name/schema and semantic inputs.
- Pass: protocol facts and final result agree.

## S10 Sub-agent parent/child execution and aggregation

- Preconditions: delegation is sufficient and a real child-agent lifecycle is available.
- Input: decomposable tasks with varied child objectives.
- Expected path: parent delegates -> child Run executes -> result returns through Interaction ->
  parent aggregates.
- Forbidden path: inline fake child result or hidden direct function call.
- Database/Audit/Trace: parent/child identities, lifecycle, correlation, and aggregation are clear.
- Artifact: child outputs retain provenance into the parent result.
- Restart/integration: parent and child survive restart; real model/capabilities are used.
- Reverse case: unsuitable task remains with Main Agent or clarifies.
- Anti-hardcoding: random child count/objectives and result order.
- Pass: aggregate is complete without losing child failure facts.

## S11 Cron and Webhook create or resume Runs

- Preconditions: real Scheduler/Worker and authenticated HTTP ingress.
- Input: timezone-aware schedule and signed, replayable Webhook events.
- Expected path: trigger -> Interaction Gateway -> create/resume Task/Run -> Runtime.
- Forbidden path: scheduler/Webhook directly executes Process or business functions.
- Database/Audit/Trace: trigger identity, deduplication, missed-run/concurrency policy, and Run facts.
- Artifact: outputs use ordinary persistence.
- Restart/integration: real HTTP and Scheduler/Worker restart; default overlap policy is `skip`.
- Reverse case: invalid signature, duplicate event, and expired input fail or deduplicate safely.
- Anti-hardcoding: random schedule, timezone, event ID, and payload object.
- Pass: exactly intended Runs are created/resumed.

## S12 Explicit failure without fabricated success

- Preconditions: intentionally missing or failing real conditions.
- Input: unreachable provider/tool, invalid result, exhausted strategy, or forbidden action.
- Expected path: alternative/clarify when valid, otherwise durable explicit failure.
- Forbidden path: mock-success, fallback-success, placeholder Artifact, or converted failure.
- Database/Audit/Trace: failure category, attempted paths, terminal counts, and incomplete side
  effects are truthful.
- Artifact: no fabricated output; valid earlier facts remain identifiable.
- Restart/integration: failed state reconstructs after restart; real failure is exercised.
- Reverse case: when a valid capability exists, the same class of task succeeds normally.
- Anti-hardcoding: vary missing dependency, wording, and expected error category.
- Pass: user-visible result and durable facts both state failure.
